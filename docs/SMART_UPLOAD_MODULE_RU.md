# Smart Upload Module (next version) - архитектура и пайплайн

Документ описывает отдельный модуль для "умной" загрузки PDF и связанных файлов без риска поломать текущий TDO-контур.

## 1. Цели модуля

- Автоматически извлекать шифр документа и ключевые поля из PDF.
- Формировать структуру хранения по шаблону на основе шифра.
- Привязывать к PDF связанные файлы (DOCX, XLSX, DWG, ZIP и т.д.).
- Давать пользователю шаг подтверждения распознанных полей перед финальным сохранением.
- Работать асинхронно, чтобы не блокировать основной backend.

## 2. Пример из предоставленного PDF

Для файла `IMP-FD-00-00-IN-LST-011-00.pdf` извлекаются поля:

- `full_cipher`: `IMP-FD-00-00-IN-LST-011-00`
- `project`: `IMP`
- `phase`: `FD`
- `unit`: `00`
- `title_code`: `00`
- `discipline`: `IN`
- `doc_type`: `LST`
- `serial`: `011`
- `revision`: `00`
- `title_text`: `Fire Detection System Device List`

Рекомендуемая целевая структура:

- `IMP/FD/IN/LST/011/00/` - папка ревизии
- внутри:
  - `IMP-FD-00-00-IN-LST-011-00.pdf`
  - связанные файлы этой же поставки

## 3. Рекомендуемая архитектура (отдельный микросервис)

Компоненты:

1. `tdo-backend` (существующий):
   - принимает загрузку;
   - создает задачу на обработку;
   - отдает статус пользователю.
2. `smart-upload-api` (новый):
   - API для запуска/повтора обработки;
   - хранение результата распознавания и уверенности.
3. `smart-upload-worker` (новый):
   - парсинг PDF, OCR fallback, RegExp/правила;
   - нормализация полей и формирование путей.
4. `redis`:
   - очередь задач.
5. `s3/minio`:
   - исходные файлы, промежуточные и итоговые артефакты.
6. `postgres` (можно общий или отдельный schema):
   - метаданные загрузок и статус пайплайна.

## 4. Пайплайн обработки

### Шаг 1. Intake

- Пользователь загружает 1 PDF + N связанных файлов.
- Backend сохраняет все в `uploads/raw/{upload_id}/`.
- Создается задача `smart_upload.process(upload_id)`.

### Шаг 2. Быстрый парсинг text-layer

- Извлекается текст из первых страниц (обычно штамп/титул).
- Запускается набор regex-правил:
  - общий паттерн полного шифра: `[A-Z0-9]{2,}-[A-Z0-9]{2,}-[A-Z0-9]{2,}-[A-Z0-9]{2,}-[A-Z0-9]{2,}-[A-Z0-9]{2,}-[0-9]{2,4}-[0-9]{2}`
  - дополнительные паттерны по отдельным полям (`Project`, `Phase`, `Disc.`, `Doc. Type`, `Serial`, `Rev.`).

### Шаг 3. OCR fallback

- Если confidence ниже порога (например `< 0.80`) или шифр не найден:
  - рендер страниц в изображения (200-300 DPI);
  - OCR (Tesseract/PaddleOCR/Docling OCR);
  - повтор правил извлечения.

### Шаг 4. Нормализация и валидация

- Удаление лишних пробелов/невидимых символов.
- Приведение к upper-case.
- Валидация структуры шифра по доменным правилам.
- Проверка конфликтов (например, уже есть такой `full_cipher` + `revision`).

### Шаг 5. Human-in-the-loop подтверждение

- UI показывает:
  - найденный шифр и поля;
  - confidence;
  - превью первой страницы PDF.
- Пользователь подтверждает или правит поля вручную.

### Шаг 6. Финальная раскладка файлов

- Формируется целевая папка по шаблону.
- Все файлы перемещаются из `raw` в `library`.
- Сохраняется запись в БД (кто загрузил, когда, что распознано, кто подтверждал).

## 5. Интеграция с текущим решением (без поломок)

Рекомендуемый безопасный путь:

1. В текущем frontend добавить новый пункт меню `Smart Upload (beta)`.
2. В текущем backend добавить только тонкий proxy-роут:
   - `POST /api/v1/smart-upload/jobs`
   - `GET /api/v1/smart-upload/jobs/{id}`
   - `POST /api/v1/smart-upload/jobs/{id}/confirm`
3. Основную бизнес-логику держать только в новом сервисе.
4. Включение через feature-flag `SMART_UPLOAD_ENABLED=false` по умолчанию.

Так модуль можно развернуть и тестировать отдельно, не затрагивая существующий контур ревизий/CRS/TRM.

## 6. Технологический стек (практичный вариант)

- API/Worker: Python + FastAPI + Celery + Redis.
- PDF text extraction: `pypdf` или `PyMuPDF`.
- OCR fallback: `Tesseract` (как базовый), при необходимости `PaddleOCR`.
- Object storage: текущий MinIO/S3.
- Метаданные: PostgreSQL.

## 7. Этапы внедрения (next version)

1. Подготовка контрактов API и схемы БД для upload jobs.
2. Прототип извлечения шифра на 30-50 реальных PDF.
3. Добавление OCR fallback и измерение качества.
4. UI подтверждения полей.
5. Интеграция со storage + аудит действий.
6. Пилот на ограниченной группе пользователей.

## 8. KPI качества извлечения

- Точность `full_cipher` без ручной правки >= 92%.
- Доля файлов, требующих ручного подтверждения <= 25%.
- Среднее время обработки 1 PDF <= 10 сек (без OCR), <= 45 сек (с OCR).

## 9. Что важно сделать заранее
